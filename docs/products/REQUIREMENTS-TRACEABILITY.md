# Trazabilidad de requisitos de producto

## Propósito

Esta matriz resume el estado comprobable de los requisitos principales de PULSO IRIS, SOFÍA y LUMEN en el repositorio. La evidencia se limita a archivos versionados y describe capacidades observables; no convierte propuestas, material comercial ni referencias externas en requisitos implementados.

El estado refleja el código actual. `implementado` no equivale por sí solo a una aprobación de producción: también se requieren operación, seguridad, observabilidad y criterios de aceptación adecuados al entorno.

## Estados controlados

| Estado                   | Significado                                                                                                                                            |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `implementado`           | Existe una ruta funcional de extremo a extremo respaldada por persistencia o contratos internos y pruebas automatizadas relevantes.                    |
| `parcial`                | Existe una parte funcional, pero falta una integración, aislamiento, validación operativa o criterio esencial para cerrar el requisito.                |
| `simulado`               | El comportamiento se representa mediante un simulador o ledger interno; no existe todavía la integración externa real.                                 |
| `demo sintética`         | La capacidad puede demostrarse únicamente con datos sintéticos y no acredita operación con información real.                                           |
| `pendiente`              | No existe una implementación ejecutable suficiente para considerar iniciada la capacidad.                                                              |
| `bloqueado por decisión` | La implementación u operación está detenida intencionalmente hasta que exista una decisión explícita de producto, seguridad, cumplimiento u operación. |

## Regla de actualización

Un estado solo debe avanzar cuando la misma entrega incorpore evidencia versionada y verificable. Una pantalla, contrato, semilla o migración aislada no basta para declarar una capacidad `implementado`. Cada cambio debe actualizar la brecha o sustituirla por el nuevo criterio de cierre, sin incluir datos personales, rutas locales ni material no versionado.
